# AI Agent Instructions & Project Context

Этот файл содержит ключевую информацию об архитектуре, технологическом стеке и жестких правилах написания кода для данного проекта.

## 1. О проекте (Project Overview)
- **Описание:** Разработка In-Memory блочной аналитической базы данных с JIT-компиляцией SQL-запросов для выполнения на GPU.
- **Главная цель:** Максимальная производительность при выполнении аналитических запросов (типа Star Schema Benchmark) за счет генерации слитых ядер (Kernel Fusion) и минимизации аллокаций памяти.
- **Примеры:** В test/queries/example содержаться примеры для запросов SSB написанные вручную, они являются эталоном. В test/queries/sql содержаться SQL запросы для SSB. В test/queries/result содержится вывод примеров из test/queries/example, используй их для сравнения выходных данных. 

## 2. Технологический стек (Tech Stack)
- **Язык:** C++17 (строго).
- **GPU Framework:** SYCL 2020 (используем AdaptiveCPP).
- **Сборка:** CMake.
- **Зависимости:** `linenoise` для CLI, `hyrise/sql-parser` для построения AST.

## 3. Архитектурные паттерны (Architecture & Patterns)
- **SOLID & ООП:** Код должен быть строго разделен на слои (Presentation -> Execution -> JIT Compiler -> Storage).
- **RAII:** Вся память (особенно USM / видеопамять) управляется через RAII-обертки (например, `DynamicDeviceBuffer`). Ручные вызовы `sycl::malloc_device` и `sycl::free` вне классов-менеджеров запрещены (не касается аллокаций внутри сгенерированного кода).
- **JIT Pipeline:** AST -> Logical Plan -> Optimizer -> Physical Plan -> JIT Code Generation -> Dynamic Compilation (.so) -> `dlopen/dlsym` Execution.
- **Единый ABI:** Все сгенерированные функции имеют фиксированную сигнатуру `extern "C" void execute_query(db::ExecutionContext* ctx)`.

## 4. ЖЕСТКИЕ ОГРАНИЧЕНИЯ (CRITICAL CONSTRAINTS)
При написании кода ты **обязан** соблюдать следующие правила:

1. **Никаких заглушек (NO STUBS):** Весь код должен быть написан полностью, готов к компиляции и выполнению. Не используй `// TODO: implement this` или `...`. 
2. **Безопасность типов и переполнения (Type Safety):** - Всегда учитывай возможность Integer Overflow.
   - Используй явные касты `(unsigned long long)` при умножении больших чисел. Достаточно кастануть левый операнд.
3. **Производительность (Performance):**
   - Минимизируй использование системных вызовов в критическом пути.
   - Используй преаллоцированные буферы и паттерн High-Water Mark (с запасом емкости).
   - Применяй Predicate Pushdown: фильтры ВСЕГДА должны стоять до ресурсоемких операций (например, хеш-джойнов).
4. **Синхронизация GPU:** - Очередь `sycl::queue` всегда `in_order()`. Избегай лишних вызовов `q.wait()`.
   - Обязателен `q.wait()` **только** перед освобождением памяти или копированием на хост.
5. **Тестирование** - Всегда выполняй глубокое тестирование проделанной работы. Все исходные SSB запросы лежат в test/sql. Эталонные (ручные) версии в test/example. После внесения изменений прогоняй тестируй через утилиту db_cli. По мере необходимости добавляй/изменяй тесты в test/test_optimizer.cpp
Все эталонные результаты запросов для факторо масштабирования 10 можно найти в test/queries/result/scale10
## 5. Формат ответов (Output Format)
- Всегда предоставляй план перед началом работ
- Перед написанием кода кратко объясни архитектурный подход (1-2 абзаца).
- Если изменение затрагивает несколько файлов, явно указывай пути к файлам перед блоками кода.
- Если ты видишь архитектурный изъян в моем запросе (нарушение SRP, утечку памяти, лишний хардкод), **сообщи мне об этом до написания кода** и предложи альтернативу.
- Если требуется рефакторинг кодовой базы сообщи мне об этом.
