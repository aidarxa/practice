Перевести наш In-Memory JIT-движок (работающий на SYCL) с устаревшей модели "Pull/Late Materialization" на передовую конвейерную Push-модель.

Ключевое архитектурное требование: Мы строим надежную СУБД промышленного уровня. Поэтому для любых разрывателей конвейера (Pipeline Breakers), где итоговый размер данных неизвестен (например, построение Multi-value Hash Table для джойнов 1-to-N или промежуточная материализация фактов), мы отказываемся от Bump Allocators. Движок должен генерировать Двухпроходную сборку (Two-Pass Allocation) с использованием эксклюзивной префиксной суммы (Exclusive Prefix Sum), чтобы всегда точно рассчитывать требуемую память.

Вот пошаговый план (Roadmap) перевода движка на новую архитектуру:
Шаг 1: Смена интерфейса AST-узлов (От Pull к Push)

Сейчас JITOperatorVisitor просто обходит дерево и последовательно пишет код в stream. Для Push-модели узлы AST должны общаться друг с другом, передавая потоки данных.

Добавь в архитектуру генератора два логических метода для каждого узла:

    produce(JITContext& ctx) — Метод, который инициирует генерацию данных (открывает цикл).

    consume(JITContext& ctx, const std::vector<std::string>& active_vars) — Метод, который принимает сгенерированные данные снизу и встраивает их в свою логику.

Как это работает (на примере Scan -> Filter -> Agg):

    Движок вызывает AggNode->produce().

    AggNode ничего не генерирует сам, он просит ребенка: child->produce().

    Запрос доходит до TableScanNode. Тот начинает писать ядро: for (int i = 0...) { загрузка переменных; ....

    Внутри этого цикла TableScanNode вызывает метод родителя: parent->consume(ctx, {v1, v2}).

    Управление возвращается вверх. FilterNode вставляет if (v1 > 10) { и зовет своего родителя parent->consume(...).

    Наконец, AggNode вставляет atomic_add(...).

    Вызовы возвращаются вниз, закрывая скобки }.

Шаг 2: Модернизация Hash Table (Поддержка 1-to-N через Two-Pass)

Чтобы OR-джойны и Inner Joins корректно размножали строки, хеш-таблица на фазе Build (в crystal/join.h) должна научиться хранить несколько значений по одному ключу.

Вам нужно реализовать Multi-value Hash Table (MHT), память для которой выделяется через двухпроходную сборку:

    Структура MHT:

        Directory: [Key, Offset, Count] — по ключу находим, где лежат данные и сколько их.

        Payload Chunk: Сплошной массив со всеми данными. Его размер заранее неизвестен!

    Двухпроходная фаза Build (Генерация кода):
    Вместо одного ядра, генератор для фазы Build создает два ядра:

        Pass 1 (Count Kernel): Потоки обходят таблицу, вычисляют хеш и делают atomic_fetch_add к счетчику совпадений в массиве d_counts.

        Prefix Sum (Host): Генератор вставляет в execution.cpp вызов библиотечной функции (например, oneapi::dpl::exclusive_scan или std::exclusive_scan), которая превращает d_counts в массив d_offsets. Точный размер Payload Chunk теперь известен!

        Pass 2 (Write Kernel): Потоки снова обходят данные, берут свой точный offset и безопасно пишут данные в Payload Chunk без коллизий и перерасхода памяти.

Шаг 3: Переписывание HashJoinNode (Суть размножения)

Здесь происходит магия декартова размножения "на лету". HashJoinNode разделяет запрос на два конвейера.

В методе produce() (Фаза Build):
Он генерирует двухпроходное SYCL-ядро для левого потомка (описано в Шаге 2). На этом первый конвейер заканчивается (Pipeline Breaker).

В методе consume() (Фаза Probe):
Сюда снизу (от TableScan фактов) приходят переменные.
Генератор пишет код, который ищет ключ в Directory хеш-таблицы, получает Offset и Count, и запускает вложенный цикл.
C++

void HashJoinNode::consume(JITContext& ctx, ...) {
    // Генерируем код для Probe
    ctx.probe_kernel << "int offset = 0, count = 0;\n";
    ctx.probe_kernel << "ProbeMultiHT(" << ht_name << ", " << key_var << ", &offset, &count);\n";
    
    // Вложенный цикл - ЭТО И ЕСТЬ РАЗМНОЖЕНИЕ СТРОК НА GPU!
    ctx.probe_kernel << "for (int j = 0; j < count; ++j) {\n";
    
    // Извлекаем payload для конкретного совпадения
    ctx.probe_kernel << "    int val = payload_chunk[offset + j];\n";
    
    // ПУШИМ СТРОКУ ДАЛЬШЕ РОДИТЕЛЮ (встраиваем код Агрегации)
    parent->consume(ctx, ...); 
    
    ctx.probe_kernel << "}\n"; // Закрываем цикл размножения
}

Если у нас сложное условие OR, HashJoinNode сгенерирует несколько таких блоков (несколько Probe + циклов for), и родительский consume (Агрегация) будет вызван внутри каждого из них, гарантируя честную семантику SQL.
Шаг 4: Изоляция конвейеров (Pipeline Breakers)

Ваш JITContext должен научиться понимать, что один SQL-запрос может породить несколько sycl::queue.submit(...) (несколько ядер).

Для этого вводятся Pipeline Breakers:

    TableScan, Filter, Math — это пайплайны. Они живут в регистрах.

    HashBuild и Aggregate — это разрыватели.

Когда генератор доходит до разрывателя, который требует неизвестного объема памяти (как наш MHT Build), он делает следующее:

    Завершает текущий C++ код ядра (закрывает скобки Count Kernel).

    Регистрирует в execution.cpp выделение промежуточных массивов и вызов алгоритма Prefix Sum на хосте.

    Выделяет глобальный массив (malloc_device) точного размера.

    Открывает новое ядро (Write Kernel) для записи материализованных данных.

Шаг 5: Оптимизатор и Интеллектуальный выбор стратегий

С Push-моделью ваш оптимизатор (optimizer_rules.cpp) заживет новой жизнью.

    Спуск OR-предикатов: Вы сможете спускать OR-предикаты прямо в HashJoinNode.

    Выбор Hash Table: Оптимизатор решает, нужно ли размножение:

        Если условие простое (EQ) и ключ в каталоге помечен как UNIQUE (PrimaryKey) -> генератор выдает сверхбыстрый 1-to-1 Perfect Hash Table (один проход, без циклов размножения).

        Во всех остальных случаях (или при OR) -> генератор выдает 1-to-N Multi-value Hash Table (через двухпроходную сборку и вложенные циклы consume).
